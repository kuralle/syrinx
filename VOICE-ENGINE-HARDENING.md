# Voice Engine Hardening Notes

Date: 2026-05-28

## Current Verdict

Syrinx should keep the websocket-first engine, but the transport must be explicit about audio format and lifecycle. The active production contract is PCM16 mono at 16 kHz into the kernel, Smart Turn gated STT finalization, streamed agent text into TTS, and binary PCM output with sideband lifecycle metadata.

The latest hardening change makes the websocket layer enforce that contract:

- `ready` now advertises input/output sample rates, encoding, and channel count.
- Browser and telephony websocket servers attach their input handlers immediately on connection and buffer bounded early frames until the session is ready, so audio sent immediately after WebSocket `open` and carrier `start`/first `media` frames are not lost during session initialization.
- JSON and enveloped binary audio frames can include `sampleRateHz`; the server normalizes PCM16 into the configured engine rate.
- Browser websocket input now rejects a `sampleRateHz` change inside the same `contextId`, matching production STT stream expectations that one logical audio stream has a stable source format. New contexts can still use a different declared source rate.
- Odd-byte PCM16 payloads are rejected before they reach STT/VAD.
- Each binary TTS frame is preceded by `tts_chunk` metadata with turn id, sequence, sample rate, byte length, and duration.
- Browser and smoke clients send input sample-rate metadata and use `tts_chunk` for turn attribution.
- Non-telephony websocket assistant audio now uses the `syrinx.audio.v1` envelope by default (`SYRXA1\n` magic + little-endian JSON header length + JSON header + PCM payload), carrying turn id, sample rate, sequence, encoding, channel count, duration, and byte length in the same frame as audio. Inbound binary clients may use the same envelope, and enveloped input now requires a valid positive-integer `sampleRateHz` plus well-formed numeric metadata so malformed frames cannot be silently resampled at the wrong rate. Raw inbound PCM16 remains accepted at the advertised input sample rate. Raw outbound PCM is now an explicit `binaryAudioEnvelope: false` opt-out.
- Browser client audio utilities now test 48 kHz and 44.1 kHz capture down to 16 kHz PCM16, clamp Float32 samples before conversion, preserve JSON audio frame encoding for compatibility, and make `sendFloat32Audio()` use turn-scoped binary envelopes with explicit `sampleRateHz`.
- The browser runtime capture smoke now drives the real review console under headless Chrome with fake microphone input and verifies encoded `getUserMedia`/`AudioContext` frames are sent as `syrinx.audio.v1` envelopes, arrive at the websocket server, and that an enveloped assistant-audio frame from the server is decoded by the browser, scheduled for playback, and cleared on interruption without playback errors.
- The browser review console defaults to continuous listening once the microphone is started. A browser-side energy gate starts capture contexts with 400 ms of pre-speech audio to avoid silent phantom turns; VAD/Smart Turn and Deepgram provider finalize decide end-of-turn instead of the UI promoting push-to-talk. The browser now decodes default enveloped assistant audio before playback.
- The Twilio Media Streams adapter is now a separate websocket endpoint. It accepts only JSON text frames, validates PCMU/8 kHz/mono on `start`, converts inbound Twilio `media.payload` to engine PCM16/16 kHz, frames outbound assistant audio back to paced 20 ms PCMU `media` messages, sends Twilio `mark` after the paced audio batch drains, records carrier mark acknowledgements, sends Twilio `clear` on engine interruption, and treats provider `stop`, abrupt websocket disconnect, queued-output overflow, and outbound send-buffer refusal as terminal discard boundaries by cancelling queued playout, emitting recorder truncation, and recording `twilio.stop_playout_cleared_ms`, `twilio.disconnect_playout_cleared_ms`, `twilio.overflow_playout_cleared_ms`, or `twilio.send_buffer_playout_cleared_ms`. Twilio `mark_sent` and `clear_sent` metrics are tied to successful websocket sends, so a backpressure close cannot produce false playback-control evidence.
- The Telnyx-style Media Streaming adapter applies the same provider boundary for another carrier shape: JSON text frames, `start.media_format` validation, inbound PCMU/8 kHz or L16/16 kHz decode, PCM16 engine ingress, explicitly configured `stream_bidirectional_codec` mirror for paced outbound audio, Telnyx-shaped `media`/`mark`/`clear`, mark acknowledgement metrics, and terminal `stop`/disconnect/overflow/send-buffer handling that cancels queued playout, emits recorder truncation, and records the matching `telnyx.*_playout_cleared_ms` metric. Telnyx `mark_sent` and `clear_sent` metrics are tied to successful websocket sends, so a backpressure close cannot produce false playback-control evidence.
- The SmartPBX AI Provider adapter applies the local SmartPBX wire contract at `/media-stream`: `start`/`media`/`dtmf`/`hangup` JSON, `g711_ulaw`/8 kHz, little-endian `pcm16`/24 kHz, or Opus/48 kHz input and output, paced outbound `media` carrying `callId` plus `accountId`, local queued-audio discard on interruption, terminal `hangup`/`stop`/disconnect/overflow/send-buffer handling that cancels queued playout, emits recorder truncation, and records the matching `smartpbx.*_playout_cleared_ms` metric. Opus support uses a tested codec dependency and flushes partial assistant PCM on `tts.end` as a padded 20 ms Opus frame. SmartPBX's supplied document does not define `mark` or `clear`, so it does not send speculative interruption controls.
- Websocket endpoints now include lifecycle guards for production transport quirks: bounded early-frame buffering covers delayed session startup, heartbeat pings detect half-open connections, a bounded send-buffer guard closes slow clients with `1013` before assistant audio accumulates without backpressure, the next outbound frame is counted before send so one large frame cannot exceed the ceiling by itself, oversized inbound frames close with `1009`, strict base64 validation keeps malformed media out of STT/VAD, and telephony `stop`/`hangup`/disconnect/overflow marks the provider stream terminal so late media, queued outbound audio, and recorder artifacts cannot leak across teardown.
- Browser websocket sessions now have stable resumable session ids with an in-memory retention window, so short reconnects can continue the same `VoiceAgentSession` instead of forcing an agent/session restart.
- The recorder now mirrors transport interruption semantics: barge-in emits a recorder truncation packet before TTS clear, assistant recording trims queued audio past the wall-clock playback point instead of retaining audio the user never heard, owned write streams tolerate high-volume concurrent writes without listener warnings, and `manifest.json` records sample rates, byte counts, chunk counts, truncation count, and artifact paths.
- Deepgram STT follows provider semantics: session-long websocket, provider `Finalize`, `is_final` transcript buffering, provider text-frame `KeepAlive` during idle/post-turn playout, and provider `CloseStream` only during session shutdown. Transcript release happens only after Pipecat approves the boundary and Deepgram closes the provider segment stream with `speech_final` or `from_finalize`. A cached fallback is used only when a provider final does not arrive after an explicit finalize request. It does not reconstruct provider finals with language-specific local text heuristics. Malformed provider JSON, Deepgram `type:"Error"`/`err_*` frames, failed audio sends, and unexpected websocket close frames are surfaced as `stt.error`; documented `NET-*` websocket closes are categorized as recoverable connection failures while `DATA-*` closes are fatal invalid-input failures. Provider-boundary audio sent metrics are tied to successful websocket sends, so a closed or closing provider socket cannot inflate delivered-audio evidence.
- Once barge-in interrupts an assistant context, the core session and websocket transports treat that context as terminal for generation and playout. Late LLM deltas, late LLM completion, late TTS audio, and late TTS end signals are ignored or measured rather than spoken, so an asynchronous provider cancellation cannot reopen speech, emit terminal playback marks, or extend recorder evidence after the engine has yielded to the user. This matches the Pipecat/LiveKit/Rapida invariant that interrupted output queues are discarded rather than allowed to leak back into playout or recorder evidence.
- Cartesia TTS follows provider websocket context semantics: server-side auth uses the `X-API-Key` header rather than query-string keys, each utterance maps to a stable `context_id`, `tts.done` sends the empty terminal continuation, and interruption sends Cartesia's documented context cancel request instead of an empty flush workaround. Provider error frames, malformed provider messages, websocket errors, provider closes while contexts are active, and failed initial or terminal text sends are surfaced as context-scoped `tts.error` packets so the speech engine fails visibly rather than leaving a hung TTS context. Active-context accounting is tied to successful provider sends, so `tts.done` cannot flush a context that never reached Cartesia and later interruptions cannot cancel a context whose terminal flush already failed. Cancelled Cartesia contexts now also suppress late provider `data` and `done` frames, closing the stale-audio race that can otherwise follow a provider-side cancel.
- The session TTS path now coalesces streamed LLM deltas into sentence-complete text before synthesis, using `Intl.Segmenter` when available and a conservative terminal-punctuation fallback. Raw agent deltas are still emitted for UI/transcripts. At provider completion, any remaining buffered tail is flushed to TTS and recorded as `tts.final_tail_flushed`, matching Pipecat/LiveKit-style end-of-response flush behavior instead of dropping provider-completed text.
- The AI SDK Gemini bridge now records provider finish reasons and refuses to mark a turn successful when the provider ends with `length`, `content-filter`, `error`, `tool-calls`, `other`, or no finish metadata. This caught a live Gemini `MAX_TOKENS` truncation; the university profile now uses a larger voice token budget while keeping finish validation active.
- The live recorder coherence smoke now preserves provider STT text and uses local Whisper only as an independent recorder-audio audit. The latest live Cartesia run passed all three university turns with non-empty user/assistant recorder audio, per-turn WAV exports, zero truncations, coherent local Whisper transcripts, spoken TTS text tracked separately from raw Gemini deltas, avg STT final after audio end at 907 ms, and avg first Cartesia audio after agent text at 423 ms.
- The latest interactive websocket smoke passed three real provider turns with default enveloped assistant-audio output, avg STT final after speech end at 1,363 ms, avg first Cartesia audio after agent text at 373 ms, and avg speech-end to first assistant audio at 4,859 ms.
- The latest 24-turn longform websocket smoke passed after the Deepgram/Pipecat handshake change with avg STT final after speech end at 2,180 ms and avg Gemini TTS first audio after agent text at 10,450 ms. The longform bottleneck remains Gemini TTS, not websocket transport or STT finalization.
- Websocket and emulated Twilio, Telnyx, and SmartPBX smoke runs now emit a shared run `manifest.json` for transport, sample-rate, byte-count, duration, latency, and quality-gate audit data. Schema v2 separates websocket wire bytes from decoded/normalized PCM bytes, so PCMU and Opus artifacts preserve both transport payload size and duration-bearing PCM evidence. The smoke artifact writer validates schema version, byte totals, duration math, required compressed-audio provenance, and required telephony carrier-relative latency fields before writing. Telephony emulators support `SYRINX_EMULATED_NETWORK_PROFILE=clean|jittery|bursty` and record `maxInboundMediaGap`, so adapter evidence can distinguish ideal localhost pacing from jittery carrier media arrival. Package-level telephony tests also cover delayed-session startup buffering and terminal provider lifecycle: early carrier `start`/`media` is retained until the session is ready, late media after Twilio/Telnyx `stop`, SmartPBX `hangup`, abrupt websocket disconnect, queued-output overflow, or outbound send-buffer refusal is ignored, already queued outbound media/control work is cancelled, recorder output is truncated, and discard metrics are delivered on the critical route under blocked main dispatch.
- The carrier review server (`review:telephony`) now exposes Twilio, Telnyx, and SmartPBX websocket paths on one HTTP server and wires real carrier calls into the same live university-support session used by browser review. It also serves Twilio `<Connect><Stream>` TwiML and carrier config JSON for Telnyx/SmartPBX. Local HTTP preflight validates server startup and setup endpoints. The live-provider adapter smoke (`smoke:telephony-university-live`) now drives each carrier websocket shape with the university fixture through real Deepgram/Gemini/Cartesia and recorder output, supports clean/jittery/bursty inbound media timing via `SYRINX_TELEPHONY_NETWORK_PROFILE`, waits for paced carrier playout before hangup, writes decoded carrier-boundary inbound/outbound WAV files, runs local Whisper over both directions, and passes under jittery timing with zero recorder truncations plus non-empty carrier Whisper transcripts for Twilio, Telnyx, and SmartPBX. Completing the carrier requirement still needs public TLS plus real provider/sandbox calls.
- A Twilio outbound carrier-call smoke now exists for that real-call gap. It uses Twilio's REST Call API with `To`, `From`, and the live `/twilio/twiml` URL, polls the Call resource to terminal status, completes the call on poll timeout by default, and writes a baseline artifact. It is intentionally separate from local adapter smokes because Twilio `completed` proves carrier call setup and audio transfer, while the review server recorder/log artifacts prove the websocket media stream behavior.
- A Telnyx outbound carrier-call smoke now exists for the same gap. It uses Telnyx Call Control `POST /v2/calls` with `stream_url`, `both_tracks`, `stream_bidirectional_mode: rtp`, provider-selected bidirectional codec, `stream_establish_before_call_originate`, and `send_silence_when_idle`, then sends the provider hangup command after a bounded dwell window. As with Twilio, Telnyx command acceptance is only one layer of proof; media timing remains grounded in the review server logs and recorder artifacts.
- The websocket audio contract and default `syrinx.audio.v1` binary assistant-audio envelope are now documented in `docs/websocket-audio-protocol.md`.

## Evidence From Reference Engines

ElevenLabs Speech Engine uses one websocket per conversation, handles STT/TTS while the customer server owns LLM logic, and explicitly manages lifecycle, turn-taking, interruption, and streamed LLM output. That supports our split: keep conversation behavior pluggable, but make the speech transport and lifecycle deterministic.

AssemblyAI Voice Agent API exposes a single realtime websocket that streams user PCM audio in and agent PCM audio out, with explicit events for session readiness, speech started/stopped, user transcript deltas/finals, reply started/audio/done, agent transcript, tool calls, and session errors. This directly informed the sideband `ready` and `tts_chunk` metadata contract.

Pipecat's FastAPI websocket transport separates text/binary serialization, has connection lifecycle callbacks, optional session timeout, fixed-size audio packetization, and clears buffered output on interruption. Its Smart Turn implementation uses local VAD plus ML endpointing, keeps pre-speech context, caps the model window to the last 8 seconds, and resamples to the model's expected 16 kHz when needed. Pipecat's OpenAI LLM service streams raw text and explicitly leaves coalescing, for example full sentences for TTS, to consumers; Pipecat's text aggregator flushes remaining text when the LLM response ends. Our Smart Turn path already matches the main architectural idea; the missing pieces were transport-level format enforcement and sentence-aware TTS input with end-of-response flush.

LiveKit Agents keeps STT pumping alive across handoffs, tracks VAD and STT timing separately, persists the first speech-start timestamp across multiple VAD bursts in one logical turn, supports preemptive generation, and records end-of-turn delay/transcription delay. Its STT stream rejects mixed input sample rates and resamples only from an explicitly known frame rate to the needed STT rate; Pipecat's audio buffer processor likewise resamples input/output frames from declared frame sample rates into the target buffer rate. Syrinx mirrors that boundary by accepting raw PCM only at the advertised rate and requiring `syrinx.audio.v1` frames to declare valid sample-rate metadata before resampling. LiveKit also ships sentence tokenizer streams, including buffered sentence streams for TTS-facing consumers. Syrinx websocket smoke collectors now follow the timing invariant by preserving first speech start and updating speech end to the latest boundary for a logical turn.

Rapida normalizes telephony audio before the speech pipeline: 8 kHz telephony codecs are converted/resampled to 16 kHz LINEAR16 for input, and output is resampled/paced back to 20 ms telephony frames. That reinforces the server-side normalize-at-ingress rule instead of trusting the browser or telephony provider.

Pipecat's Twilio serializer follows the same boundary: deserialize Twilio μ-law media into pipeline PCM, serialize assistant PCM back into Twilio `media`, and serialize interruption as Twilio `clear`. Syrinx now mirrors that separation while keeping the provider adapter outside the core engine.

Pipecat and LiveKit both avoid burst-sending generated audio that can no longer be interrupted accurately. Syrinx telephony adapters now queue outbound audio as 20 ms playout frames, bound queued duration with `maxQueuedOutputAudioMs`, clear unsent frames locally on barge-in, and only send carrier marks after the paced batch has actually drained.

Pipecat, LiveKit Python, LiveKit JS, and Rapida all preserve provider STT text rather than applying language-specific post-hoc transcript reconstruction. Syrinx follows that boundary; quality smokes can detect provider divergence against fixtures, but the runtime should not rewrite multilingual transcripts with local English heuristics.

Rapida's recorder trims the system track when interruption clears queued playback. Syrinx now follows the same invariant: recordings are evidence of what should have been heard, not just what TTS generated.

Rapida's telephony processors and LiveKit/Pipecat frame abstractions keep frame size, sample rate, encoded payload, decoded audio, and timing as separate concerns. Syrinx smoke artifacts now follow that same evidence boundary: compressed carrier payload size is recorded as wire bytes, while duration and coherence checks use decoded PCM byte counts.

Cartesia's websocket documentation is context-oriented: a single connection can multiplex utterance contexts, generation responses carry `context_id`, context completion is signaled with `continue:false`, and cancellation is a first-class `{ context_id, cancel:true }` request. Syrinx now follows that provider boundary and keeps URL logs free of API keys on all maintained Cartesia websocket callers.

Modal's low-latency Pipecat write-up reinforces that voice agents are long-running stateful orchestrators with concurrent loops, local VAD/turn detection, and interruption logic; it calls out bidirectional persistent WebSockets for low-latency bot-to-service STT/TTS links and measures voice-to-voice latency from user speech end to first audible bot response. Syrinx follows the same shape: provider WebSockets stay persistent, local VAD/Smart Turn owns the boundary, and interrupted contexts are terminal for later provider output.

## Latency Direction

The public low-latency guidance is consistent:

- Keep websocket connections persistent for STT/TTS.
- Probe websocket connections with heartbeat ping/pong and fail closed when clients stop reading.
- Bound outbound audio buffering; dropping or closing a bad transport is better than allowing unbounded latency and memory growth.
- Pace telephony output at real media-frame cadence; otherwise provider buffers receive uninterruptible bursts that make barge-in and recorder evidence inaccurate.
- Measure telephony adapter latency relative to first and last inbound media frames, not only process start. The emulator manifests now expose `firstOutboundMediaAfterLastInbound` and `maxInboundMediaGap`, so live carrier timing can be compared against the local transport floor while accounting for carrier-media jitter.
- Avoid extra provider/network hops; colocate audio, inference, and call control when possible.
- Use local VAD/turn detection to bound endpointing instead of waiting for generic provider silence.
- Stream LLM text into TTS and flush at turn end; TTS buffering thresholds trade latency against quality.
- Measure stage latencies separately: speech end to STT final, STT final to first LLM text, first LLM text to first TTS audio, and speech end to first assistant audio.

## Remaining Hardening Work

- Live-test the Twilio, Telnyx, and SmartPBX adapters against carrier/sandbox streams using `review:telephony`. Current emulator/unit coverage, live-provider adapter smokes, and local review-server preflight prove the adapter wire contract, live STT/LLM/TTS path, recorder flush, setup endpoints, and teardown state machine, not real carrier timing. SmartPBX specifically needs confirmation of whether it supports any playback clear/interrupt event; the supplied protocol document does not specify one.
- Keep recorder and smoke artifact manifests aligned as the schema evolves. Current smoke schema v2 explicitly distinguishes `wireByteLength` from `decodedPcmByteLength` and has a validator in the write path; any recorder schema extension should preserve that distinction for compressed transport captures.
- Extend resumable session semantics to telephony adapters only if a provider supports reconnecting the same call stream safely.
- Treat fixture-specific transcript term checks as diagnostics only. Production STT hardening should stay on provider-boundary invariants: complete audio delivery, declared sample rate/encoding, provider `Finalize` provenance, final/interim event timing, and recorder-vs-provider artifacts for review.
