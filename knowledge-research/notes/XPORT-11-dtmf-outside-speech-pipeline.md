**Claim (one line):** DTMF tones from telephony keypads must be detected and handled *outside* the STT→LLM speech path — if they leak into a transcript they pollute the agent's input and trigger unintended responses.

**Detail.** Deepgram: *"DTMF tones should be detected explicitly and handled outside the speech pipeline so they do not pollute transcripts. Use DTMF sparingly and deterministically. Numeric shortcuts such as 'Press 0 for an operator' should route cleanly into call-control logic while preserving conversational continuity for speech interactions."* (deepgram-ebook ~752–758). The architecture rule is: the speech pipeline (VAD → STT → agent) never sees DTMF; instead the transport layer detects the tones and fires discrete call-control events (route-to-operator, transfer, terminate). This is a separation-of-concerns pattern — DTMF is a control signal, not conversational content.

Pipecat handles caller DTMF **in-pipeline**: Twilio's serializer deserializes a `dtmf` WebSocket event into an `InputDTMFFrame` (`twilio.py:267-271`), which flows through the pipeline as a `SystemFrame` (`frames.py:1330`) and is consumed by a `DTMFAggregator` frame processor (`dtmf_aggregator.py:29`). What Pipecat lacks is acoustic *in-band* DTMF detection from the media stream — caller DTMF arrives as out-of-band carrier events, not decoded from audio. Pipecat also bundles a standalone audio utility (`audio/dtmf/`) for **synthesizing** DTMF playback to the caller: it defines a `KeypadEntry` enum covering 0–9, `*`, `#` (enum at `types.py:17-47`; the "dual-tone multi-frequency (DTMF) keypad entries" phrasing is in the module docstring at `types.py:8`) and provides `load_dtmf_audio()` — an async function that loads pre-recorded `.wav` tones (e.g., `dtmf-1.wav` through `dtmf-pound.wav`), resamples to the target sample rate (default 8000 Hz), and caches in memory (`utils.py:35-71`). Deepgram also notes that call termination and transfer "must be handled explicitly" — when a call ends or is redirected, the media stream should close immediately and overlapping speech avoided during transitions (ebook ~759-762).

**Prior-art divergence.** Pipecat provides DTMF *playback* tools (WAV files + resampling utilities) plus in-pipeline handling of provider-signalled DTMF, but no in-band DTMF *detection* from audio — it expects the telephony provider (Twilio/Vonage/Telnyx) to signal DTMF out-of-band via their WebSocket protocol. LiveKit handles DTMF too: its beta workflows collect caller DTMF (`beta/workflows/dtmf_inputs.py` — `GetDtmfTask`, timeout, `#` to stop) and send DTMF (`beta/tools/send_dtmf.py` — `publish_dtmf`), with telephony/IVR examples. Rapida also handles received DTMF in its SIP pipeline (`voice-ai/api/assistant-api/sip/pipeline/control.go` `handleDTMFReceived`, routed via `dispatcher.go`'s `DTMFReceivedPipeline`, with `case "dtmf"` in its Telnyx/Exotel WebSocket handlers) — it is not document-service only. Cloudflare is a bridge (Twilio/Telnyx → Durable Object) and delegates DTMF to the carrier protocol. Deepgram's rule — "detect and handle outside the speech pipeline" — is *not* universally followed: Pipecat's `DTMFAggregator` deliberately converts DTMF into a `TranscriptionFrame` and feeds it into the LLM context path (`dtmf_aggregator.py:7-11`), mixing DTMF with the transcript stream.

**Implication for Syrinx.** Handle DTMF as a transport-layer concern: detect via carrier events (Twilio `dtmf` message), route to call-control logic, and ensure the audio buffer *excludes* DTMF-tones from the frame stream sent to VAD/STT. Bundle Pipecat's DTMF WAV assets for synthesizing keypress tones when the agent prompts for keypad input.
