# XPORT — Audio Transport (Map of Content)

## Core problem
Voice-to-voice latency and reliability are decided at the audio transport layer: how audio is carried (WebSocket vs WebRTC), in what encoding (PCM linear16, µ-law/A-law, Opus) and sample rate (8k telephony, 16k STT, 24k TTS, 48k WebRTC), how it is framed (10–50 ms chunks), buffered for smooth playback (~100 ms jitter), serialized onto each provider's wire (Twilio Media Streams), and kept alive. Mistakes here — an idle stream, an over-buffered TTS burst, a missing `streamSid`, a needless transcode — surface upward as clipped words, laggy barge-in, and turn-taking failures (deepgram-ebook line 522–523).

## Narrative
Start with the **transport choice** [[XPORT-01-ws-vs-webrtc]]: WebRTC for the lossy client link, WebSocket for server↔server and telephony carriers — both coexist in one bot (Modal's split). Inside the pipeline everything is **int16 mono PCM** with stage-specific rates [[XPORT-02-canonical-pcm-sample-rates]] (16k in, 24k out, 48k WebRTC, 8k telephony). The transport edges **resample** between the link rate and the pipeline rate, using a *stateful* streaming resampler to avoid chunk-boundary clicks [[XPORT-03-resampling-ingress-egress]]. The telephony edge is special: 8 kHz **µ-law** (G.711), where the prime directive is to **avoid transcoding** when the provider speaks µ-law natively [[XPORT-04-mulaw-telephony-path]] — and the WebRTC edge is **Opus**, decoded to PCM at ingress [[XPORT-09-opus-webrtc-codec]]. Audio moves in small **frames of 10–50 ms** [[XPORT-05-frame-chunk-sizing]], re-chunked so barge-in can cut playback mid-utterance; on the playout side a small **jitter buffer (~100 ms)** plus real-time send-pacing smooths network variance [[XPORT-06-jitter-buffer-playback]]. On the wire to Twilio, audio is a JSON envelope of base64 µ-law that must echo `streamSid` and emit `clear` on interruption [[XPORT-07-twilio-media-streams-serialization]]. Finally the transport stays alive by emitting **continuous silence** on idle and watching for stalls [[XPORT-08-transport-keepalive]].

Three cross-cutting concerns complete the transport picture. **Acoustic echo cancellation (AEC)** [[XPORT-10-acoustic-echo-cancellation]] is the WebRTC-level defence that lets the mic stay open during barge-in — without it, the agent hears itself and self-interrupts. **DTMF tones** [[XPORT-11-dtmf-outside-speech-pipeline]] must be detected and routed outside the speech pipeline so keypad input never pollutes transcripts. And **packet loss concealment / Opus FEC** [[XPORT-12-packet-loss-concealment-opus-fec]] lives entirely in the WebRTC media engine below the pipeline's visibility — no clone configures it in application code, but it determines whether jitter becomes stutter.

## Open questions / gaps
- **µ-law passthrough end-to-end:** Deepgram says skip transcoding when the provider speaks µ-law (ebook line 719), but Pipecat's pipeline is PCM-only and always transcodes. Does Syrinx's STT/TTS accept µ-law natively? If so, a passthrough serializer mode is a free latency win on telephony. (unverified for Syrinx providers)
- **Resampler quality vs latency:** Pipecat defaults soxr `VHQ`; Cloudflare uses naive linear interpolation. No clone measures the latency delta of VHQ→QQ — worth benchmarking for the 8k telephony path where fidelity is already capped.
- **Jitter buffer target:** Deepgram ~100 ms, LiveKit `queue_size_ms=200`. No source pins the optimal tradeoff vs barge-in responsiveness; depends on link (WebRTC vs WS).
- **Keepalive semantics:** clones use app-layer silence + session-timeout; none documents WS ping/pong intervals explicitly. Confirm the underlying `websockets`/aiortc defaults match our stall-detection budget.
- **Frame size default:** Deepgram/Vapi favor 20 ms; Pipecat output defaults 40 ms, LiveKit 50 ms. Pick and justify Syrinx's egress frame size against barge-in latency.
